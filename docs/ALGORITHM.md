## Как работает семафор (человеческое объяснение)

Ниже описание без диаграмм. Только словами и примерами.

Представь, что у тебя есть общий Redis и несколько приложений, которые
должны ограничить одновременный доступ к ресурсу.

---

## 1) Что хранится в Redis

Для семафора с именем `payments` создаются два ключа:

1) `semaphore:payments:owners` (ZSET)
   - member: уникальный identifier владельца (процесс/инстанс)
   - score: время истечения этого владельца (миллисекунды)

2) `semaphore:payments:fencing` (STRING)
   - целое число, которое растет через INCR
   - это fencing token

---

## 2) Что делает acquire (получить слот)

Допустим `limit=2`, а процессов 3: A, B, C.

### Что происходит в Redis

1) Удаляются просроченные владельцы (ZREMRANGEBYSCORE).
2) Считается текущее число владельцев (ZCARD).
3) Если владельцев меньше limit:
   - добавляется новый владелец в ZSET с новым expires_at
   - увеличивается fencing token
   - возвращается успех
4) Если владельцев уже limit:
   - возвращается “busy”

### Blocking vs non-blocking

- `blocking=True` (по умолчанию) — цикл ожидания с шагом `retry_interval`
  до получения слота или до `acquire_timeout`.
- `acquire_timeout=None` — ждать бесконечно.
- `blocking=False` — одна попытка и мгновенный ответ `busy`, без ожидания.

Важно: `with Semaphore(...)` всегда использует `blocking=True` и будет
ждать слот; если задан `acquire_timeout`, то при его истечении будет
исключение `AcquireTimeoutError`.

### Пример

- A делает acquire -> получает слот (token=1)
- B делает acquire -> получает слот (token=2)
- C делает acquire:
  - если blocking=False -> сразу “busy”
  - если blocking=True -> ждет и пробует снова

---

## 3) Как работает ожидание (blocking)

У нас **нет глобальной очереди в Redis**.
Ожидание — это локальный цикл в процессе:

```
попробовать acquire
если занято:
    sleep(retry_interval)
    попробовать снова
```

Поэтому в DEBUG логах `waiting=1` — это **локальный счетчик** только в этом
процессе, а не глобальное число ожидающих во всей системе.

---

## 4) Release (освободить слот)

Когда процесс закончил работу, он вызывает `release()`:

1) Останавливается heartbeat.
2) В Redis удаляется его identifier из owners.
3) Локальное состояние сбрасывается.

Если вызвать `release()` без успешного `acquire()` -> ошибка.

---

## 5) Refresh и Heartbeat (продление TTL)

### Refresh

`refresh()` продлевает TTL владельца:
1) Redis проверяет, что identifier еще есть и не истек.
2) Если все нормально -> продлевает.
3) Если истек или отсутствует -> возвращает False (lock lost).

### Heartbeat

Когда слот получен, запускается фоновый heartbeat, который регулярно
вызывает `refresh()` и продлевает TTL.

Если heartbeat не работает (процесс умер, сеть пропала), TTL истекает,
и слот освобождается сам.

---

## 6) Lock lost (потеря блокировки)

Сценарий:

- Процесс считает, что держит слот.
- Но его TTL истек или запись удалили.
- refresh возвращает False.

Что дальше:
- Если strict_mode=False: логируется предупреждение, но код продолжает работать.
- Если strict_mode=True: поднимается LockLostError сразу.

Это полезно для критичных систем, где нельзя продолжать работу после потери блокировки.

---

## 7) Fencing token (защита от гонок)

Fencing token — это число, которое растет при каждом успешном acquire.
Даже при повторном acquire тем же владельцем token становится больше.

### Зачем это нужно

Представим:
- Процесс A получил token=10, но завис и не успел завершить работу.
- Процесс B получил token=11 и продолжил.

Если ресурс (например, БД) принимает только самый большой token,
то старые операции с token=10 можно безопасно игнорировать.

---

## 8) Пример полного цикла

limit=2, процессы A, B, C:

1) A acquire -> успех, token=1
2) B acquire -> успех, token=2
3) C acquire -> занято, ждет
4) A release -> слот освободился
5) C acquire -> успех, token=3

---

## 9) Время: клиентское или серверное

По умолчанию используется время клиента (`time.time()`).
Можно включить `use_server_time=True`, тогда время берется у Redis (`TIME`).

Зачем это нужно:
- Если у разных машин часы расходятся, серверное время надежнее.
- Минус: лишний сетевой RTT.

---

## 10) Sync vs Async

Один и тот же объект `Semaphore` нельзя использовать одновременно
в sync и async режимах.

Пример неправильного использования:
```
sem.acquire()   # sync
await sem.aacquire()  # async -> ошибка
```

Это защищает от гонок внутри одного объекта.

---

## 11) Метрики и логирование

### Метрики

Метрики считаются **внутри процесса**, не глобально.
Если процессов много — Prometheus должен суммировать их сам.

### Логи DEBUG

Логи вида:
```
Semaphore 'payments' usage 2/2
Semaphore 'payments' full 2/2; waiting=1
```

- `usage 2/2` — всего слотов занято в Redis
- `waiting=1` — локально в текущем процессе
